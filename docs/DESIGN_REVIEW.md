# SupaStrength — Design Review (data model verification)

Generated by the multi-agent design pass (session 000). Three independent
reviewers (spec-coverage, engine-correctness, integrity/RLS) audited the
synthesized schema; a repair pass folded the findings back in. This file is
the record of what was checked and fixed.

**Totals:** 22 findings — 6 blocker, 5 major, 11 minor. All blockers/majors resolved in `DATA_MODEL.md` + `9999_init.sql`; minors addressed where reasonable.

---

## 1. [BLOCKER] §3 Progression state / progression_state (DATA_MODEL §3.5) vs SPEC lines 76-78 [O-5a]
- **Lens:** spec coverage
- **Problem:** SPEC locks: 'Working weight is shared per (routine, exercise)' AND 'Rep-scheme / rep targets can differ per workout-entry while weight stays shared.' The model stores current_rep_target, current_rep_range_low/high, and current_set_count on progression_state, which is keyed unique(routine_id, exercise_id) — i.e. ONE shared row per exercise across the whole routine. If squat is prescribed 5x5 in Workout A and 3x8 in Workout B (same routine), there is exactly one progression_state row, so its current_rep_target / current_set_count cannot represent both. Only weight is supposed to be shared; rep/set live state is being shared too, directly contradicting the locked 'rep targets can differ per workout-entry' clause. The DATA_MODEL §6 double-progression walkthrough makes this concrete: it advances current_rep_* on the single shared row, which would force every workout-entry using that exercise to the same live rep target.
- **Fix:** Split the live state so weight is shared per (routine, exercise) but rep/set progression state is per workout-entry. Either (a) keep only current_weight (+ weight-pipeline counters: pipeline_step_index, step_completion_counter, failure_counter, consolidation_counter where they pertain to weight) on progression_state(routine_id, exercise_id), and move current_rep_target/range/set_count plus any rep/set-dimension step indices into a per-workout_entry state row (e.g. progression_entry_state keyed by workout_entry_id, or routine_id+workout_entry_id); or (b) re-key the rep/set portion to (routine_id, workout_entry_id). Document which counters follow the shared-weight line vs the per-entry rep/set line, since pipelines can target either dimension.

## 2. [BLOCKER] Scenarios (d) double progression & (e) shoulders: per-entry rep/set state vs shared row
- **Lens:** progression-engine correctness
- **Problem:** progression_state is keyed (routine_id, exercise_id) and holds the rep/set live state: current_rep_target, current_rep_range_low/high, current_set_count, pipeline_step_index, step_completion_counter. The spec explicitly says 'Rep-scheme / rep targets can differ per workout-entry while weight stays shared' (SPEC §3, O-5a). Concretely: squat in Workout A can be 5x5 straight while squat in Workout B is 3x8-12 double-progression, sharing ONE climbing weight. But there is exactly one progression_state row for (routine, squat), so there is only one current_rep_target / one current_set_count / one pipeline_step_index. You cannot encode two different rep ladders for the same exercise on the same shared weight line. The double-progression and shoulder rep/set ladders therefore collide the moment the same exercise appears in two workouts with different rep schemes — the rep state of A overwrites B's on every completion, and the single pipeline_step_index cannot point into two different pipelines at once.
- **Fix:** Split state by dimension: keep current_weight (+ weight-pipeline counters) on the (routine, exercise) row, but move rep/set live state and their step indices to a per-(progression_state, workout_entry) child table (e.g. progression_entry_state keyed (routine_id, workout_entry_id)) so each entry tracks its own current_rep_target/current_set_count/pipeline_step_index/step_completion_counter while reading the shared current_weight. Update DATA_MODEL §6 walkthroughs (d) and (e) accordingly. Weight stays shared; reps/sets become per-entry as the spec requires.

## 3. [BLOCKER] Scenario (i) chained failure: repeat-up-to-3-then-deload has no current-position cursor
- **Lens:** progression-engine correctness
- **Problem:** failure_responses is an ordered chain (position, response_type, repeat_limit, amount) and the only runtime counter is progression_state.failure_counter (one integer). For '[{repeat, repeat_limit:3}, {deload_pct, amount:10}]' the engine must know it is currently sitting on the repeat row vs the deload row, and must reset/advance deterministically. With a single failure_counter and no current_failure_response_index, the rules are ambiguous: when failure_counter reaches 3, does the engine apply the deload on the 4th failure and then reset failure_counter to 0 and go back to the repeat row (infinite loop of repeat-3-then-deload), or stop? The DATA_MODEL prose says 'reset failure_counter' but the schema cannot record which chain position is active, so a multi-step chain longer than [repeat, deload] (e.g. repeat-3, deload-10%, repeat-3, deload-5lb) is not representable. There is also no recorded reset of failure_counter on a SUCCESS — once a deload happens and the lift succeeds, nothing in the schema says failure_counter returns to 0.
- **Fix:** Add progression_state.current_failure_response_index integer not null default 0 (cursor into failure_responses.position) alongside failure_counter (which counts repeats within the current response). Define explicitly: on failure, increment failure_counter; when failure_counter > repeat_limit of the current response, apply the next response, advance current_failure_response_index, reset failure_counter to 0; on a fully successful entry, reset both failure_counter and current_failure_response_index to 0. Document the success-reset in §6. This makes arbitrary-length chains deterministic.

## 4. [BLOCKER] Scenario (g) consolidation: desired pre-rounding increment is never persisted, so the trigger is unrecoverable
- **Lens:** progression-engine correctness
- **Problem:** The consolidation trigger (SPEC §6, Q-C) is 'when forced rounding makes the actual jump LARGER than the desired increment.' Detecting it requires comparing desired_next_weight (current_weight + step.amount, pre-rounding) against the plate-rounded actual_weight. The schema stores neither the desired pre-rounding weight nor the realized rounded weight on progression_state — current_weight is described as already the rounded shared weight, and plate solutions are explicitly never stored. So at session-commit time the engine can compute the comparison in-memory, but there is no row that records 'we wanted +5, plates forced +10' for audit reconstruction or for re-deriving consolidation_counter if it is lost. More importantly, the schema gives no field distinguishing the rounded weight actually prescribed from the ideal line, so a later session cannot tell whether the current_weight is a consolidation-held rounded value or a normal value — the only signal is consolidation_counter>0, which conflates 'holding because of oversized jump' with any other hold.
- **Fix:** Persist the ideal (un-rounded) running weight separately, e.g. progression_state.target_line_weight numeric, advanced by the pure step math, with current_weight being its plate-rounded realization. Consolidation fires when round(target_line_weight) - previous current_weight > step.amount. Record the oversized jump in the existing audit_log gap_workout row's before/after jsonb (desired vs forced). This makes the trigger reconstructable and decouples the ideal line from plate reality.

## 5. [BLOCKER] Section 11 — updated_at triggers on videos / progress_photos
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** The trigger loop in section 11 includes 'videos' and 'progress_photos' in its table list, but neither table has an updated_at column (videos: ...created_at, expires_at; progress_photos: ...created_at, expires_at). set_updated_at() executes `new.updated_at := now()`, which references a nonexistent record field. Every UPDATE on these two tables will fail at runtime with a plpgsql 'record has no field updated_at' error. This breaks the documented paid-tier use case of extending videos.expires_at / progress_photos.expires_at per row.
- **Fix:** Remove 'videos' and 'progress_photos' from the updated_at trigger array in section 11 (they are append-mostly with created_at/expires_at only), OR add `updated_at timestamptz not null default now()` to both table definitions. Removing them from the loop is the smaller change and matches their column shape.

## 6. [BLOCKER] FK on-delete — RESTRICT to custom exercises/workouts blocks account deletion
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** progression_state.exercise_id, workout_entries.exercise_id, and session_entries.exercise_id all use `references exercises(id) on delete restrict`; rotation_workouts.workout_id uses `references workouts(id) on delete restrict`. Custom exercises and workouts are user-owned (user_id ... on delete cascade). When auth.users deletes a user, Postgres fires all dependent cascades in one statement with no guaranteed ordering; if the custom exercise/workout row is reached before its still-referencing child (also being user-cascaded), RESTRICT fires immediately (RESTRICT is not deferrable) and aborts the entire user deletion. This contradicts the model's `auth.users(id) on delete cascade` contract and breaks right-to-delete. (Referencing a global seed exercise is fine because seeds are never user-deleted; the failure is specifically for custom exercises/workouts.)
- **Fix:** For FKs that can point at user-owned rows, either (a) change `on delete restrict` to `on delete cascade` where losing the child with the parent is acceptable (rotation_workouts when a workout is hard-deleted; the model already prefers soft-delete via archived_at, so a real delete should cascade), or (b) use `on delete no action deferrable initially deferred` so the integrity check is deferred to commit, by which point the cascading child rows have also been removed. Keep RESTRICT only where the referenced row can never be a user-owned row.

## 7. [MAJOR] §7 Equipment-ceiling behavior 'auto_switch_reps' (SPEC lines 216-218) vs progression model
- **Lens:** spec coverage
- **Problem:** SPEC locks equipment-ceiling behavior with two options: default 'hold + warn', or 'auto-switch that lift to rep/set progression' when a weight progression exceeds what is loadable (dumbbells past 25, barbell past 320). The enum value ceiling_behavior='auto_switch_reps' exists on equipment_preferences and workout_entries.ceiling_behavior_override, but nothing in the model expresses WHAT rep/set progression to switch to, or records that a lift HAS auto-switched. There is no flag on progression_state (e.g. 'weight ceiling reached, now progressing reps') and no link to a fallback rep/set pipeline or target. The engine cannot persist the switched mode, so the behavior is named but not actually modeled.
- **Fix:** Add state to capture the switched mode — e.g. a column on progression_state such as ceiling_reached boolean / active_dimension or progression_mode ('weight' | 'reps_fallback'), and define how the rep/set fallback is parameterized (reuse the resolved pipeline's rep/set steps, or a dedicated fallback). Without this the auto_switch_reps enum is inert.

## 8. [MAJOR] Scenario (d) double-progression loop: on_cap='loop' has no target step, and reset on the weight step is mis-placed
- **Lens:** progression-engine correctness
- **Problem:** The §6 walkthrough encodes double progression as step1 {reps, cap rep_count=12, on_cap:next_step, reset:reps_to_base} -> step2 {weight, +5, on_cap:loop}. Two problems. (1) on_cap='loop' is defined in the check constraint but has no semantics for WHERE it loops to — there is no loop_target_position column, so 'loop' is implicitly 'back to step 0', which is undocumented and fragile if the pipeline has >2 steps or the weight step is not last. (2) The walkthrough puts reset:reps_to_base on step1's on_cap:next_step, but the actual reps-to-8 reset must happen AFTER the +5 is applied (when transitioning step2 -> step1), otherwise the ramp from 8->12 happens at the OLD weight and the reset on step1's exit fires before the weight bump. As written, step1 resets reps to base on the SAME transition that should be moving to heavier weight, so the engine resets reps to 8 then immediately is on the weight step at 12-rep state — order is ambiguous. The cap on step2 is also cap_type:none with on_cap:loop, which the constraint allows (on_cap loop with cap none) but means 'loop after every single application,' i.e. +5 once then immediately back to reps — correct by luck, but undocumented.
- **Fix:** Either add progression_steps.loop_target_position (default 0) to make 'loop' explicit, or document that loop returns pipeline_step_index to 0. Move reset:reps_to_base onto the WEIGHT step (step2) so reps reset to base happens together with the +5 bump, and state in §6 the exact evaluation order (apply weight delta -> apply reset -> set pipeline_step_index to loop target). Add a check that on_cap='loop' is only valid when reset is set or cap_type<>'none' so a degenerate infinite no-op loop is rejected.

## 9. [MAJOR] every_n counter (scenarios c, and any step with every_n>1) is not reset across step transitions or failures
- **Lens:** progression-engine correctness
- **Problem:** step_completion_counter drives every_n (bench +5 every 2nd = every_n:2). It is a single counter on progression_state and is incremented per completion. Two gaps: (1) When a pipeline advances to a different step (double progression, shoulders), the same step_completion_counter keeps its value, so a step2 with every_n:2 would inherit step1's parity — there is no per-step counter or documented reset on step transition. (2) On a failure-repeat (scenario i), the spec holds weight and re-attempts; it is unspecified whether a held/failed session still increments step_completion_counter toward the next every_n application. As written a failed bench could still tick the 'every 2nd' counter and apply +5 on the next success even though a set was missed.
- **Fix:** Either make the counter per-step (move it to a per-(state, step) child, or reset step_completion_counter to 0 on every pipeline_step_index change), and define in §6 that failed/held sessions do NOT increment step_completion_counter (only qualifying completions do). Document the every_n arithmetic precisely (apply when (counter+1) % every_n == 0).

## 10. [MAJOR] Scenario (h) scope inheritance: pipeline binding is exclusive per scope, but resolution across 3 levels is not expressible for partial overrides
- **Lens:** progression-engine correctness
- **Problem:** progression_pipelines has settings_id UNIQUE, bound 1:1 to a progression_settings row, and progression_settings has partial-unique per scope (one routine row, one workout row, one entry row). The walkthrough '+5 to whole workout, but bench +5 every 2nd' relies on the engine picking the most-specific PIPELINE (exercise -> workout -> routine). That works for a full pipeline override. But the spec says inheritance is per-setting and 'most specific wins' for EACH knob (warmup/rest/consolidation coalesce independently). Pipelines do NOT coalesce field-by-field — they are whole objects bound to one settings row. So you cannot say 'inherit the workout's weight step but override only every_n at the exercise level'; you must redefine the entire pipeline at the exercise scope. For the bench example that means duplicating the +5 step with every_n:2, which is fine, but the schema offers no way to express a partial pipeline delta, and the DATA_MODEL claims pipeline inheritance 'reuses the same scope precedence' — it reuses precedence for SELECTION, not for field-level merge, which the prose conflates.
- **Fix:** Clarify in DATA_MODEL §6 that pipelines resolve by whole-object most-specific-wins (not field merge), and that an exercise-scope override must restate the full pipeline. If field-level pipeline inheritance is actually desired, that needs a different model (e.g. step-level overrides). At minimum, document the selection-vs-merge distinction so implementers do not expect coalesce semantics for pipelines.

## 11. [MAJOR] Immutability — completed-session children are not protected
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** prevent_completed_session_mutation() only guards the `sessions` row. set_logs, session_entries, session_overrides have no immutability trigger, so after a session is 'completed' a client (under its own RLS) can still UPDATE actual_weight/actual_reps or DELETE individual set_logs/session_entries. The model and SPEC state sessions and their logged history are immutable; analytics views read these children directly, so post-hoc edits silently rewrite history and skew est-1RM/volume.
- **Fix:** Add BEFORE UPDATE OR DELETE triggers on session_entries, set_logs, and session_overrides that reject the mutation when the owning session.status = 'completed' (look up the parent via session_entry_id -> session_id, or denormalize a session_status). Allow writes only while the parent session is in_progress. Keep body_measurements mutable (correct per doc).

## 12. [MINOR] §7 Equipment ceiling for dumbbells/discrete loading (SPEC lines 213-218)
- **Lens:** spec coverage
- **Problem:** SPEC: 'Dumbbell lifts snap to the discrete owned bells (15/20/25); a weight progression caps at 25 until more are bought,' and the ceiling behavior applies to dumbbells too. The plate-calc/rounding settings (rounding_direction, micro_plates_enabled, ceiling_behavior) are framed around barbell plate math. Dumbbell ceiling detection requires the engine to know the max available bell at the active location, which is derivable from the dumbbells table, but ceiling_behavior_override lives only on workout_entries and equipment_preferences with no explicit note that it governs dumbbell-loaded exercises as well. The model supports the data but the ceiling-behavior wiring is implicitly barbell-centric.
- **Fix:** Confirm (in DATA_MODEL prose) that ceiling_behavior applies to any loaded exercise whose computed weight exceeds the max loadable for its loading_style (barbell total > plate sum, or dumbbell weight > max owned bell), and that the engine reads dumbbells for the dumbbell ceiling. No schema change strictly required.

## 13. [MINOR] §9 Strength-vs-standards 'percentile' (SPEC lines 240-243 [M6]) vs v_strength_vs_standards
- **Lens:** spec coverage
- **Problem:** SPEC M6 'Relative to standards' specifies a 'percentile vs published novice->elite standards by bodyweight/sex (needs profile: bodyweight, sex, age, height).' The view v_strength_vs_standards returns a coarse discrete band (beginner/novice/intermediate/advanced/elite), not a percentile, and joins only on sex + bodyweight bracket — age and height from user_profiles (birthdate, height_in) are present but unused. The DATA_MODEL explicitly defers percentile interpolation to the UI. This is a defensible split (band in SQL, interpolation client-side), but as written the model under-delivers the locked 'percentile' framing and ignores the age/height inputs the spec calls out.
- **Fix:** Either confirm the band-then-client-interpolate approach satisfies M6 (and note where age/height are used, if at all, since standards are typically only sex+bodyweight), or have the view also emit the bounding thresholds and bodyweight so the UI can compute a true percentile. Clarify whether age/height are genuinely required for the standards lookup or were over-specified.

## 14. [MINOR] §4 Warm-up sets interaction (SPEC lines 148-153 [O-8]) — basis=volume
- **Lens:** spec coverage
- **Problem:** SPEC locks warmup auto-generation 'only if working load exceeds a threshold' with basis default = working weight, 'configurable to volume.' The model carries warmup_threshold_basis check in (working_weight, volume) and warmup_threshold_value, but a 'volume' basis threshold has no defined operand in the schema (volume = sets x reps x weight is derived, not a single comparable scalar stored anywhere on the prescription). It is unclear what value the engine compares against for basis='volume' at prescribe time. The ramp default {0,55,70,85} correctly encodes 'empty bar -> ~55% -> 70% -> 85%.'
- **Fix:** Document how basis='volume' is evaluated (what computed scalar warmup_threshold_value is compared to at prescribe time), or note it as an engine-resolved computation. Schema is adequate; the semantics need spelling out so the locked 'configurable to volume' option is implementable.

## 15. [MINOR] §8 'Most often' tracking [M5] (SPEC line 228) — coverage by views
- **Lens:** spec coverage
- **Problem:** SPEC M5 locks 'Most often tracks workouts + exercises + muscles.' The underlying data supports all three (session_entries.workout_id for workouts, session_entries.exercise_id for exercises, exercise_muscles for muscles), but DATA_MODEL §5's enumerated analytics views cover volume, e1rm, strength, standards, and reminders — there is no view for workout/exercise frequency. This is not a contradiction (views are not claimed exhaustive and the data is present), just an unmodeled-at-the-view-layer item worth flagging so it is not lost.
- **Fix:** Add (or note as TODO) a v_frequency / 'most often' view aggregating counts of workouts, exercises, and muscles per time window, mirroring the §8 time-window toggles (7d/4wk/12wk/all).

## 16. [MINOR] Scenario (j) dumbbell snap to 15/20/25 and equipment ceiling: no per-exercise loadable-set or ceiling state
- **Lens:** progression-engine correctness
- **Problem:** Dumbbell snap is a client plate-calc concern, but the ceiling_behavior ('hold_warn' vs 'auto_switch_reps') has nowhere to record that a lift HAS hit its ceiling and (if auto_switch_reps) has been converted to rep progression. equipment_preferences.ceiling_behavior and workout_entries.ceiling_behavior_override define the POLICY, but progression_state has no flag like 'at_ceiling' or a record that the weight dimension is frozen and reps are now progressing. So for a dumbbell lift stuck at 25 with auto_switch_reps, the engine must re-derive 'we are at ceiling' from equipment every session and there is no audit/state that the switch occurred. Round up/down (rounding_direction) and micro_plates_enabled are correctly modeled.
- **Fix:** Add progression_state.weight_frozen boolean (or at_ceiling) set when the computed weight exceeds loadable max under the location's equipment; when ceiling_behavior=auto_switch_reps, the engine then drives the rep dimension and records the switch in audit_log. Document the dumbbell discrete-snap as a pure solvePlates concern over the dumbbells table (15/20/25 rows), which the schema already supports.

## 17. [MINOR] RPE rep_scheme has no progression dimension; planned vs live RPE not tracked in state
- **Lens:** progression-engine correctness
- **Problem:** workout_entries.rep_scheme allows 'rpe' with target_rpe, and session_entries.planned_rep_scheme mirrors it, but progression_steps.dimension is only weight|reps|sets — there is no RPE-driven progression, and progression_state has no current_target_rpe. None of the ten listed scenarios is RPE-based, so this is not blocking them, but the data model advertises an rpe scheme it cannot progress, which will surface as a dead option.
- **Fix:** Either document that 'rpe' scheme is prescription-only (no auto-progression, manual weight selection) in DATA_MODEL, or add an rpe handling path (target_rpe in progression_state + an autoregulation step type) if RPE progression is intended later.

## 18. [MINOR] Scenario (a/b/c) qualifying-completion definition for shared weight is ambiguous when an exercise appears twice in one gym day
- **Lens:** progression-engine correctness
- **Problem:** Progression 'advances per completed workout' (Q-B). If the same exercise appears in two workouts that are both heads of two rotations on the SAME gym day (e.g. squat in the cycling rotation AND in an always-on rotation), the shared progression_state row would be advanced twice in one session, double-incrementing current_weight and step_completion_counter. The schema does not prevent or define this; session_entries can hold two entries for the same exercise in one session, and the §6 advance logic says 'on each qualifying squat completion' without de-duping per session.
- **Fix:** Define in §6 that progression_state for a given (routine, exercise) advances at most once per session (per gym day), choosing a deterministic entry (e.g. the heaviest/last completed), or explicitly allow N advances if that is intended. Add the dedupe rule to the engine spec to avoid silent double-progression.

## 19. [MINOR] RLS FORCE — exercises and exercise_muscles
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** The model says enable RLS + FORCE on every user-owned table. The 12.4 loop applies `force row level security`, but exercises and exercise_muscles are handled separately in 12.2/12.3 with only `enable row level security` (no FORCE). Without FORCE, the table owner role bypasses these policies. Low practical risk (seed writes go through service-role/bypassrls anyway), but it is an inconsistency with the stated invariant.
- **Fix:** Add `alter table exercises force row level security;` and `alter table exercise_muscles force row level security;` after their enable statements in 12.2/12.3.

## 20. [MINOR] strength_standards uniqueness — bracket dedupe ignores bw_max
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** ux_strength_standards_bracket is unique on (lift_key, sex, coalesce(bw_min_lb,-1)). Two rows with the same lift/sex/min but different bw_max_lb would collide and one would be rejected, and conversely overlapping brackets that share a min but differ in max can't both exist. Since the lookup in v_strength_vs_standards joins on a half-open [min,max) range, the dedupe key should reflect the full bracket.
- **Fix:** Either include the max in the unique key, e.g. unique (lift_key, sex, coalesce(bw_min_lb,-1), coalesce(bw_max_lb,'infinity'::numeric)), or document that brackets are keyed by min only (one open-ended max per min). Pick the one that matches the intended dataset shape.

## 21. [MINOR] Deferrable position uniques require runtime deferral for reorders
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** The (workout_id, position), (rotation_id, position), (pipeline_id, position), (failure_rule_id, position) uniques are `deferrable initially immediate`. A single UPDATE that swaps two positions is still checked per-row mid-statement under 'initially immediate' and will transiently violate, so reordering needs the app to issue `set constraints ... deferred` (or do swaps via a temp value). This is a usage gotcha, not a defect.
- **Fix:** Note in the app/data-access layer that bulk position reorders must run inside a transaction with `set constraints all deferred`, or change these to `deferrable initially deferred` so the common reorder path works without explicit deferral.

## 22. [MINOR] SECURITY DEFINER RPCs vs FORCE RLS
- **Lens:** data integrity, RLS, and Supabase migration soundness
- **Problem:** set_default_location / set_default_barbell are SECURITY DEFINER and run as the (super-ish, bypassrls) migration owner, so FORCE RLS on locations/barbells does not constrain them; they rely solely on the in-body `user_id = auth.uid()` filters. This works in Supabase (postgres has bypassrls) but couples correctness to the definer's privilege rather than to RLS. The functions are also not idempotency-affecting, just worth flagging for the security posture.
- **Fix:** Confirm the definer role and keep the explicit `where user_id = auth.uid()` predicates (already present) as the real guard; consider adding `revoke execute ... from public` / explicit `grant execute to authenticated` so only intended callers can invoke them.

## Repair notes (from the repair pass)

All blockers and majors from the review are resolved; minors addressed where reasonable. Key structural changes:

BLOCKERS
1. O-5a weight/rep-set split: `progression_state(routine_id, exercise_id)` now holds ONLY the shared weight line + weight cursor/counters + failure cursor + ceiling flags + the ideal `target_line_weight`. New table `progression_entry_state(routine_id, workout_entry_id)` holds per-entry rep/set live state and rep/set pipeline cursor. This makes squat-5x5-in-A vs squat-3x8-12-in-B representable on one shared weight line.
2. Failure chain cursor: added `current_failure_response_index` + documented success-reset of both it and `failure_counter`. Arbitrary-length chains now deterministic.
3. Consolidation: added `target_line_weight` (ideal un-rounded line) so the oversized-jump trigger is reconstructable; documented recording `{desired, forced}` in the gap_workout audit row.
4. `updated_at` added to `videos` and `progress_photos` (instead of removing them from the trigger loop) so the paid-tier expires_at extension via normal updates keeps working and the shared trigger does not error.
5. Account-deletion FKs: `exercise_id` FKs (workout_entries/session_entries/progression_state) changed to `no action deferrable initially deferred`; `rotation_workouts.workout_id` changed to `on delete cascade`.

MAJORS
6. Ceiling auto-switch state: `progression_mode` + `weight_frozen` on progression_state; documented the auto_switch_reps path driving per-entry rep/set steps.
7. `loop_target_position` on progression_steps + a check rejecting degenerate no-op loops; documented exact double-progression evaluation order (apply weight delta -> reset reps -> loop) with reset moved onto the weight step.
8. `every_n` counter resets to 0 on step-index change (documented for both weight and rep/set cursors); failed/held sessions do not increment counters.
9. Completed-session child immutability: new `prevent_completed_child_mutation()` trigger on session_entries, set_logs, session_overrides.
10. Pipeline resolution documented as whole-object most-specific-wins (selection, not field-merge), distinct from knob coalesce.

MINORS
- FORCE RLS added to exercises + exercise_muscles.
- strength_standards unique key now includes bw_max (full half-open bracket).
- All position uniques are `deferrable initially deferred` (bulk reorder works without explicit set constraints).
- RPC execute revoked from public, granted to authenticated (purge to service_role).
- New `v_frequency` view for M5 "most often" (workouts/exercises/muscles x time windows).
- v_strength_vs_standards now emits the four bounding thresholds + bodyweight for client percentile interpolation.
- Prose added for: warmup basis='volume' operand, dumbbell ceiling scope, RPE-as-prescription-only, percentile/age-height clarification, and the once-per-session shared-weight dedupe rule.

ASSUMPTIONS / OPEN ITEMS
- Per-session dedupe of the shared weight advance is specified as engine behavior ("last completed, heaviest" driving entry) rather than a DB constraint, since the choice is policy and the schema cannot easily enforce "advance at most once per (session, exercise)" without engine cooperation. Flagged in DATA_MODEL §6.
